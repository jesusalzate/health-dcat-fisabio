Guía de Extensiones del Portal CKAN

Esta guía describe las extensiones y complementos que deben instalarse sobre una instancia de CKAN 2.11 ya operativa para replicar la funcionalidad completa del portal GVA PharmaTrace Hub de FISABIO.

Punto de partida: Se asume que CKAN 2.11 está correctamente instalado con sus servicios de soporte (PostgreSQL, Redis y Solr) funcionando. El entorno virtual de Python se encuentra en /usr/lib/ckan/default y el archivo de configuración en /etc/ckan/default/ckan.ini.

La arquitectura añadida se compone de cuatro bloques:

Tema FISABIO
Identidad visual y páginas institucionales
DCAT
Interoperabilidad con estándares europeos de salud
Scheming
Formularios de metadatos adaptados al EHDS
Harvester
Importación automática de catálogos externos

Plugins que quedarán activos

Tras completar esta guía, la directiva ckan.plugins del portal contendrá los siguientes identificadores (en el orden recomendado):

harvest ckan_harvester dcat_rdf_harvester image_view text_view datatables_view datastore fisabio_theme dcat dcat_json_interface structured_data scheming_datasets

A continuación se describe el origen y función de cada uno:

Plugin Extensión Descripción
harvest ckan_harvester dcat_rdf_harvester ckanext-harvest Importación periódica de catálogos remotos en formatos RDF/TTL. Incluye soporte DCAT nativo.
image_view text_view datatables_view CKAN core Visualizadores nativos de recursos: imágenes, texto plano y tablas interactivas.
datastore CKAN core Motor de datos estructurados que permite consultar recursos CSV mediante API.
fisabio_theme ckanext-fisabio Tema visual de FISABIO, páginas institucionales y sistema de solicitud de acceso a datos restringidos.
dcat dcat_json_interface structured_data ckanext-dcat Serialización del catálogo en RDF semántico (HealthDCAT-AP) para interoperabilidad con el Espacio Europeo de Datos de Salud.
scheming_datasets ckanext-scheming Formularios de creación de datasets basados en el esquema de metadatos HealthDCAT-AP.

Paso 1 Instalación del tema personalizado (ckanext-fisabio)

El tema de FISABIO es una extensión desarrollada a medida que encapsula la identidad visual del portal y varias funcionalidades propias. Debe instalarse en el entorno virtual de CKAN a partir de la carpeta de código fuente proporcionada.

Requisito previo: Copiar la carpeta ckanext-fisabio al servidor. La ruta recomendada es /usr/lib/ckan/default/src/ckanext-fisabio.
1 Activar el entorno virtual e instalar el tema
source /usr/lib/ckan/default/bin/activate cd /usr/lib/ckan/default/src/ckanext-fisabio pip install -e .
2 Verificar que el plugin es reconocido por CKAN
ckan -c /etc/ckan/default/ckan.ini plugin list

La salida debe incluir fisabio_theme en la lista de plugins disponibles.

Qué instala este paso: Las páginas /proyecto, /socios y /resultados, el formulario de solicitud de acceso en /solicitar-acceso, el CSS de marca FISABIO y todas las sobreescrituras de plantillas del portal.

Paso 2 Instalación de DCAT (ckanext-dcat)

ckanext-dcat es la extensión oficial de la comunidad CKAN para la publicación de catálogos de datos en formatos semánticos estándar (RDF, Turtle, JSON-LD). En este portal se utiliza específicamente con el perfil HealthDCAT-AP, que es el perfil europeo de metadatos para el sector sanitario definido en el marco del European Health Data Space (EHDS).

source /usr/lib/ckan/default/bin/activate pip install -e git+https://github.com/ckan/ckanext-dcat.git#egg=ckanext-dcat pip install -r /usr/lib/ckan/default/src/ckanext-dcat/requirements.txt
Nota: Si el servidor no tiene acceso a Internet en el momento de la instalación, puede descargarse el repositorio de ckanext-dcat por separado y subirse al servidor para instalarlo con pip install -e /ruta/local.

Paso 3 Instalación de Scheming (ckanext-scheming)

ckanext-scheming permite definir los campos del formulario de creación y edición de datasets mediante un fichero YAML de esquema, en lugar de usar los campos genéricos que CKAN trae por defecto. Se utiliza para cargar automáticamente el esquema de metadatos HealthDCAT-AP incluido en la propia extensión DCAT.

pip install -e git+https://github.com/ckan/ckanext-scheming.git#egg=ckanext-scheming pip install -r /usr/lib/ckan/default/src/ckanext-scheming/requirements.txt

Paso 4 Instalación del Harvester (ckanext-harvest)

ckanext-harvest habilita la funcionalidad de cosecha (harvesting) del portal: permite importar catálogos de datos desde fuentes externas de forma programada. El portal FISABIO lo utiliza junto al módulo dcat_rdf_harvester para poder ingerir ficheros en formato Turtle (TTL) y RDF publicados por otros sistemas sanitarios.

pip install -e git+https://github.com/ckan/ckanext-harvest.git#egg=ckanext-harvest pip install -r /usr/lib/ckan/default/src/ckanext-harvest/pip-requirements.txt
Inicializar las tablas de base de datos del Harvester

Después de instalar la extensión, hay que crear las tablas que el harvester necesita en la base de datos de CKAN:

ckan -c /etc/ckan/default/ckan.ini harvester initdb
Proceso en segundo plano: El harvester requiere un proceso de trabajo (worker) corriendo en segundo plano. Consulte el apartado de configuración más abajo para ver cómo iniciar este proceso con systemd.

Configuración del archivo ckan.ini

Una vez instaladas todas las extensiones, es necesario actualizar el archivo de configuración principal de CKAN para activar los plugins y ajustar los parámetros de cada uno. El archivo se encuentra habitualmente en /etc/ckan/default/ckan.ini.

Antes de editar: Realice una copia de seguridad del archivo original antes de modificarlo.
cp /etc/ckan/default/ckan.ini /etc/ckan/default/ckan.ini.bak

Lista de plugins activos

Localice (o añada) la línea ckan.plugins dentro del bloque [app:main] y establezca exactamente el siguiente valor:

ckan.plugins = harvest ckan_harvester dcat_rdf_harvester image_view text_view datatables_view datastore fisabio_theme dcat dcat_json_interface structured_data scheming_datasets

Conexión a bases de datos

Ajuste los valores de conexión a PostgreSQL según las credenciales de su servidor:

# Base de datos principal de CKAN sqlalchemy.url = postgresql://ckan:CONTRASEÑA_CKAN@localhost/ckandb # DataStore (lectura/escritura y solo lectura) ckan.datastore.write_url = postgresql://ckan:CONTRASEÑA_CKAN@localhost/datastore ckan.datastore.read_url = postgresql://datastore_ro:CONTRASEÑA_RO@localhost/datastore # Solr y Redis solr_url = http://localhost:8983/solr/ckan ckan.redis.url = redis://localhost:6379/1

Servidor de correo (SMTP)

El formulario de solicitud de acceso del portal envía notificaciones por correo electrónico al administrador. Es necesario configurar un servidor SMTP válido:

# Configuración de correo saliente smtp.server = smtp.gmail.com:587 smtp.starttls = True smtp.user = correo_admin@su-dominio.es smtp.password = CONTRASEÑA_DE_APLICACION smtp.mail_from = correo_admin@su-dominio.es # Dirección a la que se envían las solicitudes de acceso email_to = administrador@su-dominio.es
Recomendación de seguridad: Si usa Gmail, genere una Contraseña de aplicación desde la configuración de seguridad de la cuenta de Google (requiere verificación en dos pasos activa). No use la contraseña principal de la cuenta.

Perfil HealthDCAT-AP y esquema de metadatos

Estas directivas activan el perfil europeo de salud para la serialización RDF del catálogo y cargan el esquema de campos del formulario de creación de datasets:

# Cola de trabajo del Harvester (requiere Redis) ckan.harvest.mq.type = redis ckan.harvest.mq.hostname = localhost ckan.harvest.mq.port = 6379 ckan.harvest.mq.redis_db = 0 # Perfil RDF activo: European Health Data Space ckanext.dcat.rdf.profiles = euro_health_dcat_ap # Esquema de metadatos HealthDCAT-AP para el formulario de datasets scheming.dataset_schemas = ckanext.dcat.schemas:health_dcat_ap.yaml scheming.presets = ckanext.scheming:presets.json ckanext.dcat.schemas:presets.yaml

Reiniciar el servidor tras los cambios

Después de guardar el archivo ckan.ini, reinicie el proceso de CKAN para que los cambios surtan efecto:

sudo systemctl restart ckan # O, si usa Supervisor: sudo supervisorctl restart ckan-uwsgi

Funcionalidades incluidas en cada extensión

A continuación se describe de forma práctica qué aporta cada componente una vez configurado el portal.

Identidad visual FISABIO

El tema aplica la paleta corporativa (azul marino #003E7E y naranja #E8500A), tipografías y componentes de interfaz adaptados a la imagen de FISABIO y la GVA, incluyendo cabecera, pie de página y menú de navegación.

Páginas institucionales integradas

Las rutas /proyecto, /socios y /resultados son páginas nativas del portal (no externas) que heredan el mismo diseño que el resto de secciones, facilitando el mantenimiento y garantizando coherencia visual.

Acceso bajo demanda a datasets restringidos

Cuando un dataset se marca como restringido (campo access_rights = RESTRICTED o campo extra acceso_bajo_demanda = true), el portal muestra automáticamente un botón de solicitud en la página del dataset. El usuario rellena un formulario y el administrador recibe un correo estructurado con los datos de la solicitud, sin necesidad de que el usuario esté registrado.

Espacio Europeo de Datos de Salud (EHDS)

El catálogo de datos se publica automáticamente en formato RDF siguiendo el perfil HealthDCAT-AP. Cualquier plataforma europea del sector sanitario puede consumir el catálogo completo mediante los endpoints /catalog.rdf, /catalog.ttl y /catalog.jsonld.

Formulario de metadatos extendido

El formulario de creación y edición de datasets incluye todos los campos específicos de HealthDCAT-AP (tipo de dato de salud, nivel de acceso, legislación aplicable, período temporal, etc.) en lugar de los campos genéricos de CKAN, facilitando el cumplimiento del estándar desde el primer registro.

Cosecha automática de catálogos

Desde la sección de administración del portal es posible configurar fuentes externas (URLs de catálogos RDF, ficheros TTL, etc.) para que sean importadas periódicamente de forma automática. Los nuevos datasets aparecen publicados sin intervención manual.

Recursos visuales requeridos

El tema incluye imágenes estáticas que deben estar presentes en la carpeta pública del servidor CKAN. Todos estos archivos se copian automáticamente al instalar la extensión con pip install -e ., pero se listan aquí para referencia en caso de necesitar reemplazarlos:

Archivo Descripción
proyecto_logo.pngLogo principal del proyecto en la página /proyecto
fisabio_gva_n_b.pngLogo combinado FISABIO-GVA (versión blanca/negra)
socio_01.png — socio_04.pngLogos de las cuatro entidades socias (FISABIO, IIS La Fe, ISABIAL, FIHGUV)
financiacion_01.png — financiacion_04.pngLogos de los organismos financiadores (Convocatoria 2024 digitalización)
Nota: Estos archivos se sirven directamente desde la raíz del portal (p. ej., https://su-portal.es/proyecto_logo.png). Si necesita actualizar algún logo, basta con reemplazar el archivo en la carpeta ckanext-fisabio/ckanext/fisabio_theme/public/ y reiniciar el servidor.

Proceso de trabajo del Harvester (opcional pero recomendado)

Para que la cosecha automática funcione en producción es necesario mantener un proceso worker activo en segundo plano. La forma recomendada es configurarlo como un servicio de systemd:

1 Crear el archivo de servicio
sudo nano /etc/systemd/system/ckan-harvest-worker.service

Con el siguiente contenido:

[Unit] Description=CKAN Harvest Worker After=network.target [Service] User=www-data ExecStart=/usr/lib/ckan/default/bin/ckan -c /etc/ckan/default/ckan.ini harvester run-test Restart=on-failure RestartSec=5 [Install] WantedBy=multi-user.target
2 Activar e iniciar el servicio
sudo systemctl daemon-reload sudo systemctl enable ckan-harvest-worker sudo systemctl start ckan-harvest-worker
